Analysis:

The initial code violates the Interface Segregation Principle (ISP) because it uses a large interface BookReportService that forces classes to implement methods they do not need. This causes low cohesion, high coupling and poor maintainability due to irrelevant functionalities like email, and PDF export.
Refactoring Approach:
To fix this issue, the large interface was split into smaller more specific interfaces SalesReportGenerator, InventoryReportGenerator, EmailSender, and PDFExporter. Each interface handles a single responsibility and each class implements only the interface relevant to its functionality and avoids unnecessary implementations.
Justification:
After refactoring, each interface has a single responsibility and classes are no longer forced to implement unused methods. This improves cohesion, reduces coupling, and enhances maintainability and scalability of the system.
